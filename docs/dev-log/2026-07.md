### 決めたこと

#### Issueの作業単位

- Issueは、単独で目的と完了を判断できる作業単位とする
- 小規模な変更は、1 Issue・1コミットを目安とする
- 同じ目的を達成する一連の変更は、複数コミットに分かれる場合も1 Issueとして扱う
- 大規模な作業は親Issueと子Issueへ分け、実作業と完了判定は原則として子Issue単位で行う
- 設計、実装、テスト、docs更新が同じ成果を完成させるために不可分である場合は、工程ごとにIssueを分割しない

#### Issue本文の項目

- 必須項目は`概要`と`完了条件`とする
- `背景`、`現状`、`判断事項`、`対応方針`、`参考資料`は任意項目とする
- `参考資料`はIssue本文の末尾へ配置する
- `対象`、`対象外`、`確認項目`、`テスト項目`、`レビュー観点`は共通の独立項目にしない
- 完了条件はチェックリスト形式とし、Issueごとに具体化する
- Issue Forms調査後に、種類ごとの最低限形式と詳細形式の必要性を判断する

#### Issue Formsの採用候補

- Issue FormsをIssue Templateの第一候補とする
- 共通の最低限形式は、種類ごとに分けず1フォームへ統合する候補とする
- 任意項目は、種類ごとに使用頻度の高いものだけを配置する
- 完了条件には初期値を設定しない
- PriorityとAreaはIssue本文やIssue Formへ記載せず、GitHub Projectで管理する
- 分類別の詳細フォームでは対応するラベルの自動付与を検討する

- `docs/specs/task-backlog.md` は、GitHub Issuesへの移行が完了するまで未完了タスクの既存一覧として維持する
- `docs/specs/_drafts/task-backlog-reorganization-draft.md` は、GitHub Issuesへの移行対象、移行状況、独自IDとIssue番号の対応、統合・分割・削除候補、移行漏れを確認するための一時資料として使用する

### GitHub Issues移行手順

1. Issue本文フォーマット案を作成する
2. GitHub Project、field、labelを作成する
3. 少数のサンプルIssueを作成する
4. サンプルIssueの運用を確認し、Issue本文フォーマットを確定する
5. 必要に応じてIssue Templateを作成する
6. `task-backlog-reorganization-draft.md`を移行管理用に整理する
7. 残りの未完了タスクをIssueへ移行する
8. 件数と移行漏れを確認する
9. `task-backlog.md`の廃止または役割変更を行う

大量のIssueを一括作成せず、サンプルIssueで運用を確認してから残りを移行する。
