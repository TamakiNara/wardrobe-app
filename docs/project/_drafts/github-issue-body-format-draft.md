# GitHub Issue本文設計ドラフト

> [!NOTE]
> このファイルは、GitHub Issue本文の項目と記載粒度を確定するための設計ドラフトです。
> Issue Formsの調査とサンプルIssueでの検証が完了するまで、Issue TemplateやIssue Formsへ反映しません。
> 正本化する場合は、ドラフト固有の説明と未決定事項を整理します。

## 目的

GitHub Issue本文に記載する情報、項目の役割および記載粒度を定義する。

Issue作成時点で過度な事前調査を要求せず、作業を開始し、完了を判断するために必要な情報を記載できる形式とする。

GitHub Projectのフィールドやラベルで管理する情報を、Issue本文へ重複して記載しない。

## Issueの作業単位

### 基本単位

Issueは、単独で目的と完了を判断でき、他の変更と分けてレビューできる作業単位とする。

小規模な変更は、1 Issueを1コミットで完了できる粒度を目安とする。

ただし、同じ目的を達成する一連の変更は、複数コミットに分かれる場合も1 Issueとして扱う。

### 親Issueと子Issue

複数の独立した成果物を含む作業は、親Issueと子Issueへ分ける。

親Issueは、次の内容を管理する。

- 全体の目的
- 対象範囲
- 子Issueの関係
- 全体の進捗

実作業と完了判定は、原則として子Issue単位で行う。

一部分だけ完了し、残りを保留しても意味のある状態になる場合は、Issueの分割を検討する。

### 工程による分割

設計、実装、テストおよびdocs更新が、同じ成果を完成させるために不可分である場合は、工程ごとにIssueを分割しない。

複数機能へ影響する設計判断や、後続の実装範囲を決定する設計は、独立した`design` Issueとして扱う。

局所的な設計判断は、実装を扱う`feature`または`bug` Issueに含めてよい。

## Issue本文の基本方針

- Issue本文は、作業開始時点で分かっている情報を基に作成する
- 原因、修正箇所、更新対象ファイルが未調査でもIssueを作成できる形式とする
- Issue本文は仕様書の代わりにしない
- 現在有効な仕様と確定した設計判断は個別仕様書へ記載する
- 作業経緯、調査過程、採用しなかった案は開発ログへ記載する
- Status、Priority、Area、ラベルはIssue本文へ記載しない
- すべてのIssueへ同じ任意項目を機械的に設けない
- 不要な見出しはIssue作成時に残さない

## 必須項目

現時点の必須項目は、次の2項目とする。

### 概要

Issueで解決、実現または決定する内容を記載する。

作業方法や調査結果ではなく、Issueの目的が分かる内容とする。

### 完了条件

このIssueを完了とする状態をチェックリストで記載する。

完了条件は、単に`実装する`、`確認する`、`docsを更新する`とせず、Issueの内容に合わせて具体化する。

ただし、Issue作成時点で詳細が確定していない場合は、調査後に完了条件を更新してよい。

完了条件の入力欄には初期値を設定しない。

フォーム上の説明文で、このIssueを完了とする条件を具体的なチェックリストで記載するよう案内する。

テスト、動作確認、仕様書更新が必要な場合は、その対象と完了状態が分かる形で完了条件へ記載する。

入力例はプレースホルダーで示し、Issue本文へ自動挿入しない。

## 任意項目

次の項目は、Issueの内容に応じて追加する任意項目とする。

### 背景

Issueが必要になった経緯や、目的だけでは理解しにくい前提を記載する。

目的や概要と重複する場合は使用しない。

### 現状

現在の動作、仕様または問題が発生している状態を記載する。

新規機能など、記載すべき現在状態がない場合は使用しない。

### 判断事項

設計や仕様について決定が必要な内容を記載する。

主に`design` Issueで使用するが、他の種類でも明示的な判断が必要な場合は使用できる。

### 対応方針

採用する対応方法や、変更時に維持する制約が既に決まっている場合に記載する。

使用頻度は低い想定とし、方針が未確定の場合や、完了条件と重複する場合は使用しない。

### 参考資料

Issueへ着手する前または作業中に参照するとよい資料を記載する。

対象は、仕様書、設計資料、関連Issue、関連Pull Request、参考となるソースコードなどとする。

参考資料がない場合は使用しない。

参考資料はIssue本文の末尾へ配置する。

## 独立項目として設けない内容

次の内容は、現時点では独立した共通項目として設けない。

### 対象

修正箇所や影響範囲は、Issue作成時点では未調査の場合がある。

必要な場合は、完了条件、対応方針、参考資料または本文中の説明へ含める。

### 対象外

共通項目としては設けない。

対象範囲の明示が必要なIssueでは、概要または対応方針へ記載する。

### 確認項目・テスト項目

完了条件と重複しやすいため、共通の独立項目にはしない。

必要なテストと動作確認は、完了条件へ具体的に記載する。

### レビュー観点

Issue本文の共通項目にはせず、Pull Requestまたはレビュー時に扱う。

## Issue種類ごとのテンプレート方針

対象とするIssueの種類は次の4種類とする。

- `feature`
- `bug`
- `design`
- `docs`

共通する設計方針は維持しつつ、種類ごとに本文テンプレートを分けて検証する。

最終候補は次の5フォームとする。

- `共通` フォーム
- `feature` 詳細フォーム
- `bug` 詳細フォーム
- `設計` 詳細フォーム
- `docs` 詳細フォーム

ただし、最初から5フォームすべてを作成しない。

最初の検証対象は次の3フォームとする。

- `共通` フォーム
- `bug` 詳細フォーム
- `設計` 詳細フォーム

この3フォームで運用を確認した後、`feature` と `docs` の詳細フォームを作成するか判断する。

フォームは種類ごとに分けず、`共通` フォームへ統合する候補とする。

上記はまだ確定した実装方針ではなく、サンプルフォームで検証する候補とする。

`設計` 詳細フォームの説明文は「設計や仕様について決定が必要な事項を整理するフォームです。」とする。

## 現時点の項目構成候補

### `共通` フォーム

- 概要
- 完了条件

### `feature` 詳細形式候補

- 概要
- 背景
- 現状
- 対応方針
- 完了条件
- 参考資料

### `bug` 詳細形式候補

- 概要
- 現状
- 背景
- 対応方針
- 完了条件
- 参考資料
- 画像・ログ等の添付

### `design` 詳細形式候補

- 概要
- 背景
- 現状
- 判断事項
- 対応方針
- 完了条件
- 参考資料

### `docs` 詳細形式候補

- 概要
- 背景
- 現状
- 対応方針
- 完了条件
- 参考資料

`bug` では、発生している事象を先に確認できるよう、`現状` を `背景` より前へ配置する。

専用の添付欄が必要か、`textarea` への添付で足りるかは未決定事項として残す。

## Issue Forms調査結果

### 利用できる入力形式

Issue Formsでは、次の入力形式を使用できる。

- `textarea`
- `input`
- `dropdown`
- `checkboxes`
- `upload`
- `markdown`

各入力欄には、用途に応じて説明文、プレースホルダー、初期値、必須設定を指定できる。

Issue作成後は、フォームの入力内容がMarkdown形式のIssue本文として保存され、通常のIssueと同様に編集できる。

### 今回の項目との対応

Issue本文の各項目は、原則として次の形式で扱う。

| 項目               | 入力形式                            |
| ------------------ | ----------------------------------- |
| 概要               | `textarea`                          |
| 背景               | `textarea`                          |
| 現状               | `textarea`                          |
| 判断事項           | `textarea`                          |
| 対応方針           | `textarea`                          |
| 完了条件           | `textarea`                          |
| 参考資料           | `textarea`                          |
| 画像・ログ等の添付 | `upload` または `textarea` への添付 |

説明用の `markdown` は、入力方法や項目の使い分けを案内する場合に使用する。

### 参考資料

参考資料のプレースホルダーは、具体的なファイル例を固定せず、次の入力例とする。

```text
docs/specs/...
#123
PR #45
https://...
```

### 完了条件

完了条件には、汎用的なチェックリストを初期値として設定しない。

完了条件は空の `textarea` とし、説明文とプレースホルダーでチェックリスト形式の記載を案内する。

プレースホルダーは入力例としてのみ表示し、Issue本文の初期値として保存しない。

完了条件のプレースホルダー候補は、次の程度に留める。

- [ ] ○○への遷移が可能
- [ ] ○○の場合の動作を確認
- [ ] ○○の仕様書を更新

上記は文言の確定案ではなく、サンプルIssueで記入しやすさを確認するための候補とする。

### 任意項目

Issue Formsには、入力内容に応じて項目を表示・非表示にする条件分岐がない。

任意項目をフォームへ配置した場合、未入力可能であっても作成画面には常に表示される。

そのため、すべての任意項目を全フォームへ機械的に配置せず、Issueの種類ごとに使用頻度の高い項目だけを配置する。

### 必須設定

Issue Formsでは入力欄に必須設定を指定できる。

ただし、公式仕様上、必須設定による送信制限はpublic repositoryでのみ有効とされている。

設計上の必須項目は `概要` と `完了条件` のままとするが、リポジトリの公開範囲に応じて実際の動作を確認する。

必須設定だけに依存せず、フォーム上の説明でも記載を促す。

### ラベルとProject

Issue Formでは、フォームごとにラベル、担当者、追加先Projectを設定できる。

分類別フォームでは、対応する次のラベルを自動付与する案を検証する。

- `feature`
- `bug`
- `design`
- `docs`

ラベルは、Issue Formを使用する前にリポジトリへ作成する必要がある。

`共通` フォームでは分類ラベルを自動付与せず、Issue作成後に設定する案を検証する。

GitHub Projectへ自動追加する方法は、Issue Formの設定またはProject側の自動追加機能を候補とする。

PriorityとAreaはProjectのフィールドとして管理し、Issue本文やIssue Formへ重複して入力させない。

### Issue Formsの位置づけ

Issue FormsをIssue Templateの第一候補とする。

ただし、Issue Formsはpublic previewであり、仕様変更の可能性がある。

正式採用は、サンプルフォームとサンプルIssueで次を確認した後に判断する。

- 任意項目が多すぎないか
- 完了条件を記入しやすいか
- 空欄の任意項目が作成後のIssue本文でどう表示されるか
- 分類別フォームを分ける効果があるか
- `共通` フォームで情報が不足しないか

- `共通` フォームは、種類ごとに分けず共通で使用する候補とする
- 完了条件には初期値を設定しない
