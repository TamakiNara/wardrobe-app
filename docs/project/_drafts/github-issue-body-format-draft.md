# GitHub Issue本文設計ドラフト

> [!NOTE]
> このファイルは、GitHub Issue本文の項目と記載粒度を確定するための設計ドラフトです。
> Issue Formsの調査とサンプルIssueでの検証が完了するまで、Issue TemplateやIssue Formsへ反映しません。
> 正本化する場合は、ドラフト固有の説明と未決定事項を整理します。

## 目的

GitHub Issue本文に記載する情報、項目の役割および記載粒度を定義する。

Issue作成時点で過度な事前調査を要求せず、作業を開始し、完了を判断するために必要な情報を記載できる形式とする。

GitHub Projectのフィールドやラベルで管理する情報を、Issue本文へ重複して記載しない。

## Issueの作業単位

### 基本単位

Issueは、単独で目的と完了を判断でき、他の変更と分けてレビューできる作業単位とする。

小規模な変更は、1 Issueを1コミットで完了できる粒度を目安とする。

ただし、同じ目的を達成する一連の変更は、複数コミットに分かれる場合も1 Issueとして扱う。

### 親Issueと子Issue

複数の独立した成果物を含む作業は、親Issueと子Issueへ分ける。

親Issueは、次の内容を管理する。

- 全体の目的
- 対象範囲
- 子Issueの関係
- 全体の進捗

実作業と完了判定は、原則として子Issue単位で行う。

一部分だけ完了し、残りを保留しても意味のある状態になる場合は、Issueの分割を検討する。

### 工程による分割

設計、実装、テストおよびdocs更新が、同じ成果を完成させるために不可分である場合は、工程ごとにIssueを分割しない。

複数機能へ影響する設計判断や、後続の実装範囲を決定する設計は、独立した`design` Issueとして扱う。

局所的な設計判断は、実装を扱う`feature`または`bug` Issueに含めてよい。

## Issue本文の基本方針

- Issue本文は、作業開始時点で分かっている情報を基に作成する
- 原因、修正箇所、更新対象ファイルが未調査でもIssueを作成できる形式とする
- Issue本文は仕様書の代わりにしない
- 現在有効な仕様と確定した設計判断は個別仕様書へ記載する
- 作業経緯、調査過程、採用しなかった案は開発ログへ記載する
- Status、Priority、Area、ラベルはIssue本文へ記載しない
- すべてのIssueへ同じ任意項目を機械的に設けない
- 不要な見出しはIssue作成時に残さない

## 必須項目

現時点の必須項目は、次の2項目とする。

### 概要

Issueで解決、実現または決定する内容を記載する。

作業方法や調査結果ではなく、Issueの目的が分かる内容とする。

### 完了条件

IssueをCloseできる状態をチェックリストで記載する。

完了条件は、単に`実装する`、`確認する`、`docsを更新する`とせず、Issueの内容に合わせて具体化する。

ただし、Issue作成時点で詳細が確定していない場合は、調査後に完了条件を更新してよい。

完了条件の初期候補として、次のチェック項目を設ける案を検討対象とする。

- [ ] 目的とする変更または判断を完了する
- [ ] 必要なテストを追加または更新する
- [ ] 必要な動作確認を行う
- [ ] 必要な仕様書を更新する

これらは固定文としてそのまま使用せず、Issueごとに具体的な内容へ書き換えるか、不要な項目を削除する前提とする。

## 任意項目

次の項目は、Issueの内容に応じて追加する任意項目とする。

### 背景

Issueが必要になった経緯や、目的だけでは理解しにくい前提を記載する。

目的や概要と重複する場合は使用しない。

### 現状

現在の動作、仕様または問題が発生している状態を記載する。

新規機能など、記載すべき現在状態がない場合は使用しない。

### 判断事項

設計や仕様について決定が必要な内容を記載する。

主に`design` Issueで使用するが、他の種類でも明示的な判断が必要な場合は使用できる。

### 対応方針

採用する対応方法や、変更時に維持する制約が既に決まっている場合に記載する。

使用頻度は低い想定とし、方針が未確定の場合や、完了条件と重複する場合は使用しない。

### 参考資料

Issueへ着手する前または作業中に参照するとよい資料を記載する。

対象は、仕様書、設計資料、関連Issue、関連Pull Request、参考となるソースコードなどとする。

参考資料がない場合は使用しない。

参考資料はIssue本文の末尾へ配置する。

## 独立項目として設けない内容

次の内容は、現時点では独立した共通項目として設けない。

### 対象

修正箇所や影響範囲は、Issue作成時点では未調査の場合がある。

必要な場合は、完了条件、対応方針、参考資料または本文中の説明へ含める。

### 対象外

共通項目としては設けない。

対象範囲の明示が必要なIssueでは、概要または対応方針へ記載する。

### 確認項目・テスト項目

完了条件と重複しやすいため、共通の独立項目にはしない。

必要なテストと動作確認は、完了条件へ具体的に記載する。

### レビュー観点

Issue本文の共通項目にはせず、Pull Requestまたはレビュー時に扱う。

## Issue種類ごとのテンプレート方針

対象とするIssueの種類は次の4種類とする。

- `feature`
- `bug`
- `design`
- `docs`

共通する設計方針は維持しつつ、種類ごとに本文テンプレートを分けて検証する。

各種類について、次の2形式を候補とする。

- 最低限の項目だけを持つ形式
- 任意項目を含む詳細形式

ただし、種類ごとに最低限形式を分ける必要がない場合は、共通の最低限形式へ統合してよい。

テンプレート数は現時点では確定しない。

Issue Formsの機能と制約を調査し、フォーム上で任意項目をどのように扱えるか確認した後に決定する。

## 現時点の項目構成候補

### 共通の最低限形式

- 概要
- 完了条件

### `feature`の詳細形式候補

- 概要
- 背景
- 現状
- 対応方針
- 完了条件
- 参考資料

### `bug`の詳細形式候補

- 概要
- 背景
- 現状
- 対応方針
- 完了条件
- 参考資料

### `design`の詳細形式候補

- 概要
- 背景
- 現状
- 判断事項
- 対応方針
- 完了条件
- 参考資料

### `docs`の詳細形式候補

- 概要
- 背景
- 現状
- 対応方針
- 完了条件
- 参考資料

上記は確定テンプレートではなく、Issue Forms調査とサンプルIssueで検証する候補である。
