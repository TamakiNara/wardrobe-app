# docs 作成・更新規約

docs は、仕様、設計判断、運用上の責務を読み手が追跡できる状態に保つ。docs 間の責務、正本の配置、仕様書の構成、更新時の記載方法は、この規約に従う。

## 基本方針

docs 間の責務と正本を定め、仕様、API 契約、DB 設計、未完了タスク、作業経緯をそれぞれの正本へ記載する。同じ内容を複数の docs に記載する場合は、正本以外を参照または要約に留める。

現在有効な仕様と、未決定事項、将来候補、作業経緯を分離する。

コード上の識別子は、実装上の名称を維持する。

略語は、プロジェクト内で意味が明確でない場合、具体的な表現へ置き換える。`MVP` は文脈に応じて `初期実装範囲`、`初回対応範囲`、`最小構成` のように記載する。

## 表記・命名

### 表記言語

本文、見出し、分類名、機能名、画面名は日本語で記載する。

file path、route、API field、DB column、status 値、型名その他のコード上の識別子は、実装上の名称を使用する。

コード上の識別子は backtick で囲む。

日本語名とコード上の名称を併記する場合は、日本語名を本文上の主名称とする。

### コード上の名称

API field、DB column、file path、route、status 値、型名などの実装上の識別子は、コード上の名称として記載する。

例:

- `sale_ends_at`
- `docs/api/openapi.yaml`

コード上の名称は、無理に日本語へ翻訳しない。

### 機能名・画面名

機能名と画面名は日本語名を主に記載する。必要な場合に限り、コード上の名称や route を補足する。

例:

- 着用履歴（wear logs）
- 着用履歴詳細画面（`/wear-logs/{id}`）

route や API endpoint を説明する箇所では、実装上の名称をそのまま記載する。

## docs 間の役割

docs 間の正本責務は次のとおりとする。

- 機能仕様: `docs/specs/**`
- API 契約: `docs/api/openapi.yaml`
- API 方針: `docs/api/api-overview.md`
- DB schema・保存責務: `docs/data/database.md`
- import/export: `docs/specs/import-export.md`
- 未完了タスク、未決定事項、将来候補、優先度、依存関係: `docs/specs/task-backlog.md`
- 作業経緯、調査過程、判断理由、試したこと、採用しなかった案、次タスク: `docs/dev-log/**`
- 旧資料: `docs/archive/**`

## 機能仕様書の書き方

### 機能仕様書の役割

機能仕様書には、現在有効な仕様を記載する。

確定した設計判断、制約、他機能への影響を記載する。

現在仕様を理解するために必要な理由を記載する。

### 基本構成

すべての仕様書へ機械的に同じ章を置かず、対象仕様に必要な章だけを採用する。

基本構成の候補は次のとおりとする。

- 目的
- 対象範囲
  - 対象
  - 対象外
- 機能固有の仕様
- システム上の扱い
  - 画面
  - API
  - DB
  - ログ
- 他機能への影響
- テスト観点
- 関連 docs

見出し 2 は大分類、見出し 3 以下は詳細として扱う。細かな論点は見出し 2 で平坦に並べすぎず、関連する大分類の下へまとめる。

### 対象範囲

対象範囲は本文に記載する内容と一致させる。

対象には、この仕様書が責務を持つ内容を記載する。

対象外には、関連するが別の docs へ委譲する内容を記載する。

別の docs へ詳細を委譲する場合も、この仕様書が保証する原則と責務の境界を記載する。

### 現在仕様の書き方

関係する OpenAPI、DB、実装の整合を確認し、現在仕様として確定した内容を記載する。

調査状況や確認状況そのものは仕様本文へ記載しない。

### 詳細仕様の委譲

別の docs が詳細仕様の正本である場合、機能仕様書には、その機能が保証する原則、責務の境界、参照先を記載する。

詳細仕様を複数の機能仕様書へ重複して記載しない。

未整備の内容を、参照先に既に存在するものとして記載しない。

ログの形式、出力項目、レベル、検索方法および障害対応手順は、ログ仕様へ記載する。各機能仕様書には、その機能で記録する事象と記録目的を記載する。

### テスト観点

`テスト観点` は、仕様本文の要約ではなく、テストで確認する条件と結果を記載する。

状態遷移、境界値、不正操作、副作用、関連データの保持、他機能との組み合わせ、API と DB の整合を対象とする。

表現の分かりやすさや docs 構成の妥当性は、仕様書レビューで確認する。
