# Weather Historical Integration

気象庁データを使って、着用履歴向けの天気実績を補助取得する場合の事前調査と詳細設計メモです。
今回は実装せず、current / planned / 要再判断 を分けて整理します。

---

## 目的

- 毎日の天気手入力の負担を下げる
- 予報値とは別に、実績値を補助取得できる余地を整理する
- 個人開発アプリとして運用可能な取得経路を見極める
- 気象庁公式データに近い信頼性を優先しつつ、実装コストと運用リスクのバランスを取る

---

## current

- 手動天気登録 MVP は `weather_records` を正本としている
- 予報取得は `forecast_area_code` を使う別機能として存在する
- 予報取得はフォーム反映のみで、自動保存しない
- `source_type` / `source_name` / `source_fetched_at` を保持できる
- `was_exposed_to_rain` は未実装で、将来の着用履歴フィードバック側で扱う想定

---

## 候補比較

### 1. 気象庁 過去の気象データ検索 / ダウンロード

概要:
- 気象庁の過去観測データを検索し、CSV でダウンロードできる
- 気象官署、特別地域気象観測所、アメダス観測所を対象にできる
- 日別値、時別値、地点、項目、期間を組み合わせて取得する前提

取得しやすい項目:
- 日最高気温
- 日最低気温
- 日降水量
- 時別の天気
- 地点によっては湿度、風、日照など

向いている用途:
- 過去日の手動補助
- 既存データの後追い補完
- 管理用の確認、再取得、バックフィル

弱み:
- 画面・ダウンロード中心の導線で、安定したアプリ向け API としては扱いにくい
- CSV は多段ヘッダ、品質情報、現象なし情報、均質番号などを含みうる
- 公式にもアクセス集中や過度の自動アクセスへの注意がある
- 毎日のユーザー操作から直接使う MVP としては重い

判断:
- 公式性とデータの幅は強い
- ただし日常利用の MVP 実装先としては第一候補にしない
- 管理用 / 将来の補助経路としては有力

### 2. 気象庁 最新の気象データ CSV

概要:
- 気象庁が公開する「最新の気象データ」の CSV
- 最近 1 週間分の日別データや、今日の進行中データを扱える
- 降水・風・気温・雪ごとに CSV が分かれている

取得しやすい項目:
- 日最高気温
- 日最低気温
- 日降水量
- 1 時間降水量、24 時間降水量
- 日最大風速、日最大瞬間風速
- 積雪関連

向いている用途:
- 当日または直近数日の実績補助
- 予報値ではなく、観測ベースでの最高 / 最低気温の補助反映
- ユーザーが登録画面で「実績を取得」する導線

弱み:
- CSV であり JSON API ではない
- 文字コードが Shift_JIS、改行コードが CRLF
- 要素ごとにファイルが分かれるため、1 回の取得で必要項目が完結しない場合がある
- 天気概況のような人が見る `telop` 相当はそのままは得にくい

判断:
- 公式データを使う MVP 候補としては最も現実的
- 最高 / 最低気温、降水量の補助取得には向く
- 「天気コード」よりも「観測値の補助取得」に強い

### 3. 気象庁 アメダスデータ

概要:
- 観測所単位の観測データ
- 地域名ではなく、観測所を基準にデータを見る
- 気温、降水量、風向・風速、湿度、積雪深などを観測

取得しやすい項目:
- 気温
- 降水量
- 湿度
- 風向・風速
- 積雪深

向いている用途:
- 地点に近い観測値を補助取得したい場合
- 将来、服装フィードバックと風 / 湿度 / 雨量を結びつけたい場合
- 地域名より観測地点を明確に持てる場合

弱み:
- `forecast_area_code` とは別に、観測所コードまたは観測所識別子が必要
- 湿度や積雪はすべての地点で観測しているわけではない
- UI 上の「地域名」と観測地点が 1 対 1 とは限らない

判断:
- 実績取得の本命は「予報区域」ではなく「観測所」寄り
- 実績取得を本気で育てるなら、将来的に観測所コードの概念はほぼ必須

### 4. 第三者ラッパー API

例:
- cultivationdata.net

利点:
- JSON で扱いやすいことが多い
- アメダスや過去データへのアクセスが実装しやすい
- 観測所コードや最寄り地点検索の補助がある場合がある

リスク:
- 運営継続性、可用性、仕様変更、利用条件を自前では制御できない
- 公式データそのものではなく、第三者加工を経由する
- 個人開発では便利だが、将来の移行コストが読みづらい

判断:
- 調査・試作・比較用には有用
- 本番 MVP の第一採用先にはしない
- 使うとしても、公式経路の代替ではなく補助候補として扱う

## 気象庁データをアプリから使う現実性

### 現実的に使いやすいもの

- 直近実績の補助取得:
  - 「最新の気象データ CSV」
- 将来の過去日再取得や管理用バックフィル:
  - 「過去の気象データ検索 / ダウンロード」

### 現実的だが設計準備が必要なもの

- アメダス観測所ベースの実績取得
  - `forecast_area_code` だけでは足りない
  - 観測所コード、または緯度経度と最近傍観測所の対応付けが必要

### 現時点で避けたいもの

- UI 操作を前提とした公式ページへの過度な自動アクセス
- 非公式な scraping
- 第三者 API への早い段階での固定依存

## 取得できる項目

実績補助として価値が高いもの:
- 日最高気温
- 日最低気温
- 日降水量
- 1 時間降水量や 24 時間降水量

実績補助として将来価値が高いもの:
- 湿度
- 風向 / 風速
- 積雪関連

注意:
- `weather_code` にそのまま入れやすい「晴れ / くもり / 雨」の日概況は、予報より取りづらい
- 実績取得では、まず「気温」「降水の有無・量」を優先する方が自然
- 「地域で雨が観測された」と「自分が雨に当たった」は別概念

## 予報用コードと実績用コードの違い

結論:
- 別物として扱うべき

### 予報用コード

- 用途:
  - forecast JSON から予報を引く
- 現在の代表:
  - `forecast_area_code`
- 粒度:
  - 予報区域
- 性質:
  - 生活圏の広いエリアに対する予報

### 実績用コード

- 用途:
  - 観測所ベースの実績値を引く
- 候補:
  - アメダス観測所コード
  - または官署 / 観測地点を識別する別コード
- 粒度:
  - 観測地点
- 性質:
  - 地域名ではなく、どの地点で観測したかが重要

設計上の意味:
- `forecast_area_code` だけで実績取得を兼ねようとしない
- 将来は `forecast_area_code` と `observation_station_code` を併存できる構造にする

## weather_records への設計影響

### source 設計

現在の延長として、source は「取得手段」と「取得元」を分けて持つのが自然

推奨:
- `source_type`
  - `manual`
  - `forecast_api`
  - `historical_api`
  - 将来必要なら `observation_csv`
- `source_name`
  - `manual`
  - `jma_forecast_json`
  - `jma_latest_csv`
  - `jma_past_download`
  - `cultivationdata`
- `source_fetched_at`
  - 取得時刻

補足:
- `historical_api` は「過去実績・観測補助」全般を表す広めの分類として使える
- `observation_csv` を最初から増やすより、まず `historical_api + source_name` の組み合わせで吸収する方が柔軟

### 保存方針

推奨:
- 原則は予報取得と同じく「フォーム反映 + ユーザー確認」
- 自動保存はしない
- 取得失敗時も手入力で続けられるようにする

理由:
- 観測所の取り違えや当日途中値のリスクがある
- 実績値でも、ユーザーが登録したい内容は服装文脈で微修正が入ることがある
- source metadata を残せば、手修正後でも由来は追える

## 予報値と実績値の併存

結論:
- 併存はありうるが、同じ `weather_record` に最後に保存された値だけを置く current 設計とは分けて考える必要がある

MVP では:
- 予報取得
  - 事前入力の補助
- 実績取得
  - 後追い入力の補助
- 保存される `weather_records` は最終的に 1 レコード

将来候補:
- `forecast_snapshot` と `observed_snapshot` を別管理する
- 予報値と実績値の差分を比較できるようにする

今回は:
- docs 上では「予報補助」と「実績補助」を別概念として整理するだけに留める

## 雨観測と実体験の分離

整理:
- 地域で降水が観測された
  - 天気実績データ側の概念
- 自分が雨に当たった
  - 着用履歴フィードバック側の概念

設計方針:
- 実績取得で分かるのは、せいぜい「地域の観測所で降水があった」まで
- `was_exposed_to_rain` のような実体験フラグは `weather_records` に自動保存しない
- 雨に弱い靴やバッグの分析は、将来 `wear_logs` 側のフィードバックと組み合わせて扱う

---

## 推奨方針

### MVP 推奨

第一候補:
- 気象庁「最新の気象データ CSV」を使った実績補助

取得対象:
- 日最高気温
- 日最低気温
- 日降水量

UI 方針:
- 登録画面で「実績を取得」ボタン
- フォームに反映
- 自動保存しない
- 取得失敗時は手入力継続

理由:
- 公式データである
- 直近 1 週間の補助取得に向く
- 毎日の入力負担を下げやすい
- 過去データ検索より自動化負荷が低い

### 将来対応 推奨

段階 1:
- 実績取得用の観測所コード設計を追加
- `user_weather_locations` と観測所の紐付け方を整理

段階 2:
- アメダス / 官署ベースで湿度・風・積雪も補助取得

段階 3:
- 必要なら「過去の気象データ検索 / ダウンロード」を管理用途や再取得用途で使う

### 非推奨

- 予報区域コードだけで実績取得まで完結させる設計
- scraping 前提の実装
- 第三者ラッパー API への早期依存

---

## MVP / 将来対応の分け方

### MVP

- 実装しない段階では docs 整理のみ
- 実装する場合も、まずは直近実績の補助取得に絞る
- 取得項目は最高 / 最低気温と降水量中心
- source は `historical_api` 系で表現
- 自動保存はしない

### 将来対応

- 観測所コードの正式導入
- 緯度経度から最近傍観測所の候補提示
- 湿度 / 風速 / 積雪の補助取得
- 予報値と実績値の差分管理
- バックフィルや再取得の管理用 UI

### 要再判断

- `source_type` を `historical_api` に寄せるか、`observation_csv` を増やすか
- 実績取得 UI を「今日 / 昨日限定」にするか
- 実績取得時に `weather_code` まで自動補完するか、それとも気温 / 降水だけに絞るか
- 観測所コードの選択 UI を出すか、内部解決にするか
