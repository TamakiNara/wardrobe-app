# List Pagination Specification

一覧件数が増えたときの見せ方と、将来の API ページング方針を整理する資料。
ここでは items と outfits の一覧画面を対象に、初期表示件数、追加読み込み、カード表示内容、API の受け口を定義する。

---

## 対象画面

- アイテム一覧
- コーディネート一覧

---

## 基本方針

- 一覧は最初から全件を出さず、初期表示件数を決める
- 追加表示は無限スクロールではなく、画面下の `もっと見る` ボタンを基本とする
- API 側は将来のために、UI 表現が `もっと見る` でもページング可能な形にしておく
- UI は `もっと見る` でも、API は `page / per_page` または `cursor / limit` を受けられる形が望ましい

---

## items 一覧の表示方針

### 件数

- 初期表示は 20 件前後とする
- 画面下の `もっと見る` を押すと、さらに 20 件ずつ追加表示する
- 読み込み単位は最初と追加で大きく変えず、一定に保つ

### カードレイアウト

- 画像
- 名前
- ブランド（あれば）
- カテゴリ
- 補助情報は少なめにして、一覧での視認性を優先する

---

## outfits 一覧の表示方針

### 件数

- 初期表示は 12〜20 件程度とする
- 追加表示は items 一覧と同じく `もっと見る` ボタン方式とする
- items 一覧よりカード密度が高くなりやすいため、初期件数は 12〜20 件の範囲で調整できる前提とする

### カードレイアウト

- 名前
- 使用アイテム数
- 補助情報（季節 / TPO）
- サムネイルがあればなお良い

---

## API 方針

### 実装方針

- Backend は将来のために、ページング可能な API を前提としておく
- UI が `もっと見る` でも、内部的にはページ番号方式または cursor 方式のどちらかで追加取得できるようにする

### 候補1: page ベース

- `page` / `per_page`
- 例:
  - `page=1&per_page=20`
  - `page=2&per_page=20`

### 候補2: cursor ベース

- `cursor` / `limit`
- 例:
  - `cursor=<token>&limit=20`

### 初期推奨

- まずは items / outfits ともに扱いやすい `page / per_page` を第一候補とする
- 将来 cursor 方式へ切り替える可能性は残す
- レスポンスには追加取得可否を判断できる `has_next` 相当の情報を持たせるのが望ましい

---

## 今後の検討ポイント

- `もっと見る` の読み込み中表示と押下不可状態
- 絞り込み・検索・並び順とページングの状態同期
- 0 件時と最終ページ到達時の見せ方の違い
