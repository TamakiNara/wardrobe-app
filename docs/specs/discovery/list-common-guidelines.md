# 一覧共通仕様と今後の課題

## 1. この文書の目的

この文書は、クローゼット系 Web アプリにおける一覧画面の共通仕様について、現時点の合意内容と今後の検討課題を整理し、Codex 側の実装判断をぶらさないための引き継ぎ資料として作成する。

重視する点は以下のとおり。

- 保守性
- 設計の一貫性
- DB 整合性
- テストしやすさ
- 将来拡張

---

## 2. 現時点の結論

一覧共通仕様については、以下を採用方針とする。

1. 一覧状態は URL クエリで表現する
2. 複数フィルタは原則 AND 条件とする
3. 並び順・ページングは全一覧で共通ルールを持つ
4. 検索 UI は共通枠を持ち、画面固有条件だけ差し替える
5. 一覧の骨格は共通化し、条件項目だけ画面ごとの差分とする
6. 初期実装範囲では無限スクロールではなくページングを採用する
7. 空状態・検索結果 0 件・エラー状態を分ける
8. キーワード検索は初期実装範囲では軽量に始めるが、将来的にはタグも検索対象に含めたい

---

## 3. 一覧共通仕様（合意済み）

### 3-1. URL クエリを正本にする

一覧の検索条件・並び順・ページ番号は URL クエリに反映する。

例:

- `/items?page=2&category_id=12&season=spring&sort=updated_at_desc`
- `/outfits?tpo=work&keyword=通勤&sort=updated_at_desc`
- `/wear-logs?page=1&status=worn&date_from=2026-03-01&date_to=2026-03-31&sort=date_desc`

#### 採用理由

- リロード後も状態が維持できる
- 戻る操作に強い
- ブックマークや共有がしやすい
- UI / BFF / API の責務が整理しやすい
- デバッグしやすい

---

### 3-2. 複数フィルタは原則 AND

複数の条件を指定した場合は、すべてを満たすデータを返す。

例:

- category = tops
- season = spring
- rain_ok = true

→ 上記すべてを満たす item のみ表示する。

#### 補足

- OR 条件は初期実装範囲では採用しない
- OR は UI / API / テストが複雑化しやすいため、必要性が明確になるまで入れない

---

### 3-3. キーワード検索

初期実装範囲では軽量に始める。

#### 現時点の検索対象

- items: `name` を中心に検索
- outfits: `name` を中心に検索
- wear logs: 将来実装時に `memo` や snapshot 名称を候補とする

#### 将来方針

- **タグを導入した場合、キーワード検索の対象にタグも含めたい**
- ただし 初期実装段階ではタグ未確定のため、今すぐ必須仕様にはしない

#### 検索仕様

- 部分一致
- 前後空白は trim
- 空文字は条件なし扱い
- 複数語分割検索は初期実装範囲では不要

---

### 3-4. 並び順

全一覧で `sort` パラメータを使う。

#### sort 値候補

- `created_at_desc`
- `updated_at_desc`
- `name_asc`
- `date_desc`（wear logs 用）

#### 初期値

- items: `updated_at_desc`
- outfits: `updated_at_desc`
- wear logs: `date_desc`

#### 理由

- items / outfits は「最近触ったもの」が上に来る方が自然
- wear logs は履歴・予定を見る性格上、日付降順が自然

---

### 3-5. ページング

初期実装範囲では通常のページングを採用する。

#### 採用内容

- 無限スクロールは採用しない
- 1 ページあたり 12 件
- items / outfits / wear logs で統一する
- 将来必要があれば再検討するが、現時点では実装済みの 12件を正とする
- ページ件数のユーザー切り替えは初期実装範囲では不要

#### 理由

- URL クエリと相性が良い
- 戻る操作に強い
- テストしやすい
- 件数感を把握しやすい
- 条件変更時の挙動が明確

---

### 3-6. 共通クエリパラメータ

共通的に使用するパラメータ名を固定する。

#### 共通

- `page`
- `keyword`
- `sort`

#### 画面固有候補

- `category_id`
- `season`
- `tpo`
- `rain_ok`
- `status`
- `date_from`
- `date_to`

#### 基本ルール

- 未指定は条件なし
- `page` と `sort` の不正値は既定値へフォールバック
- 画面固有フィルタは原則バリデーション対象
- bool 系値の表現は API 側で統一すること

※ `rain_ok` の真偽値表現は、実装時に `1/0` か `true/false` のどちらかに統一すること。画面ごとに混在させない。

---

## 3-7. 現在の実装反映（2026-03 時点）

- items 一覧は `keyword / category / season / tpo / sort / page` を URL クエリと連動して扱う
- outfits 一覧は `keyword / season / tpo / sort / page` を URL クエリと連動して扱う
- BFF の GET は URL クエリを Laravel へそのまま転送し、Laravel 側で検索・並び替え・`page` を適用する
- 複数条件は AND で適用し、`sort` は現在 `updated_at_desc` と `name_asc` に対応している
- Laravel の一覧 API は `total / totalAll / page / lastPage` を meta として返し、UI で前へ / 次へのページャを描画する
- `total` は現在の検索結果件数、`totalAll` はフィルタ前の総件数である。初期実装範囲のページャ文言は `2 / 3ページ（全36件）` のように `total` のみを表示する
- 現在の items 一覧 UI は `category_id / rain_ok` ではなく `category / season / tpo` を使っている
- キーワード入力は IME 変換中に URL 更新を止め、変換確定後に検索条件へ反映する
- スマホ実機での IME 変換安定性は別途確認する

## 5. 状態別 UI ルール（合意済み）

同じ「表示件数 0」でも意味が異なるため、以下を分ける。

### 5-1. データ未登録

まだ 1 件も存在しない状態。

例:

- 「アイテムがまだありません」
- 「最初のアイテムを登録してください」

### 5-2. 検索結果 0 件

データは存在するが、条件に一致しない状態。

例:

- 「条件に一致するアイテムはありません」
- 「絞り込み条件を見直してください」

### 5-3. エラー状態

通信失敗・予期せぬ障害など。

例:

- 「一覧の取得に失敗しました」
- 「時間をおいて再度お試しください」

## 7. API 方針（合意済み）

一覧 API は Laravel の paginate ベースを前提に、レスポンス形式をできるだけ揃える。

### 想定レスポンス構造

- `data`
- `meta`
- `links`

### 利点

- フロントで共通処理しやすい
- 一覧用コンポーネントへ渡しやすい
- テストしやすい

### クエリ例

#### items

`GET /api/items?page=1&keyword=シャツ&category_id=3&season=spring&rain_ok=1&sort=updated_at_desc`

#### outfits

`GET /api/outfits?page=1&keyword=通勤&tpo=work&sort=updated_at_desc`

#### wear logs

`GET /api/wear-logs?page=1&status=worn&date_from=2026-03-01&date_to=2026-03-31&sort=date_desc`

## 8. テスト方針（一覧共通仕様に対する合意）

一覧系テストは、全境界値網羅よりも、共通仕様の骨格が守られているかを優先して確認する。

### 共通で持ちたいテスト

1. 未認証は 401
2. 自分のデータだけ返る
3. デフォルト並び順で返る
4. `page` が効く
5. `sort` が効く
6. 各フィルタが効く
7. 条件併用時に AND になる
8. 条件なしで一覧取得できる

### 画面固有で追加するテスト

#### items

- `rain_ok`
- `category_id`
- `season`

#### outfits

- `tpo`
- `season`

#### wear logs

- `status`
- `date_from`
- `date_to`

### 補足

- 業務的に意味がある条件の組み合わせを優先する
- 一覧仕様の一貫性を崩さないことを重視する

## 9. Codex 実装時の注意点

### 9-1. 共通化しすぎない

一覧の骨格は揃えるが、wear logs まで items / outfits と完全同一に寄せすぎないこと。

理由:

- wear logs は日付閲覧が中心になる可能性が高い
- 一覧の主目的が異なるため、UI の重点も異なる

### 9-2. URL クエリ同期を前提にする

- 状態を local state のみで閉じない
- 初期表示時は URL から状態復元できる形にする
- 条件変更時に `page=1` に戻すルールを忘れない

### 9-3. 不正クエリの扱いを統一する

- `sort`, `page` は既定値へフォールバック
- フィルタ値は API バリデーション方針を統一する

### 9-4. 検索対象拡張に備える

将来的にタグ検索を含める可能性があるため、キーワード検索の実装は後で対象追加しやすい構造にしておく。

例:

- 検索条件組み立て処理を分離する
- `name` のみベタ書きにしすぎない
- 将来 `tags` 結合や関連検索を追加しやすくする
