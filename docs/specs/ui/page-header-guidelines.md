# 画面ヘッダー周辺ガイドライン

## 目的

各画面のヘッダー周辺で使う要素を整理し、一覧・詳細・状態別一覧・編集画面でどこまで共通化するかを先に固定する。
全面改修の前に、差分が「仕様差」なのか「単なる揺れ」なのかを切り分け、後続の UI 改修を段階的に進めやすくすることを目的とする。

## 現状課題

- パンくず、小見出し、メイン見出し、説明文、補助リンク、件数表示、主要アクションの置き方が画面ごとに少しずつ異なる
- 一覧画面どうしでも、右上に置くのが補助リンクだけの画面と、主要ボタンを置く画面が混在している
- 詳細画面どうしでも、ヘッダーをカード化している画面と、カード外の横並びヘッダーにしている画面が混在している
- シンプルではあるが、余白・整列・役割分担のルールが弱く、やや単調で古く見えやすい
- 場当たりの個別調整を増やすと、保守しにくくなる

## 対象範囲

今回の棚卸し対象は次とする。

- item 一覧
- 手放したアイテム一覧
- item 詳細
- outfits 一覧
- 無効コーディネート一覧
- wear logs 一覧
- purchase candidate 一覧
- item / outfit / wear log / purchase candidate の代表的な編集・新規作成画面

settings 配下は後続で確認するが、まずは主要 CRUD 画面群の共通ルールを先に揃える。

## 棚卸し観点

- パンくず
- 小見出し
- メイン見出し
- 説明文
- 戻るリンク / 補助リンク
- 件数表示
- 主要アクションボタン
- ヘッダー周辺の余白 / カード / 境界線 / 横並び構成

## 画面棚卸しメモ

### 一覧画面

共通して存在する要素:

- パンくず
- 小見出し
- メイン見出し
- 説明文
- 主要アクション

揺れがある要素:

- 補助リンクの有無
- 件数表示をヘッダー内に置くか、別カードに置くか
- ヘッダー直下に空状態を置くか、filter / summary を先に置くか

現状で仕様差として扱ってよいもの:

- 主要アクションの文言や件数の意味は画面ごとに異なる
- invalid / disposed のような状態別一覧は、追加ボタンより戻るリンクを優先する

現状で単なる揺れとして扱うもの:

- 補助リンクの位置と強さ
- 件数表示カードの見せ方
- ヘッダー右側の横並びルール

### 詳細画面

共通して存在する要素:

- パンくず
- 小見出し
- 対象名をそのまま見出しにしたメイン見出し
- 右上の戻るリンク / 編集リンク

揺れがある要素:

- ヘッダーをカード化するか、カード外で置くか
- 説明文を持つか、ステータスバッジだけにするか
- 戻るリンクの文言と並び順

現状で仕様差として扱ってよいもの:

- 詳細は件数表示を持たない
- ステータスバッジは item / purchase candidate など必要な画面だけ追加する

現状で単なる揺れとして扱うもの:

- 右上アクションの並び方
- 見出し直下に補助情報を入れるルール
- ヘッダーと最初の内容カードの距離

### 状態別一覧

例:

- 無効コーディネート一覧
- 手放したアイテム一覧

共通化したい要素:

- 親一覧を含むパンくず
- 小見出しは親機能と同じ管理名
- 状態別一覧名をメイン見出しに使う
- 説明文で通常一覧と責務が違うことを明示する
- 右上は「通常一覧に戻る」を優先する
- 空状態文言は通常一覧の未登録とは分ける

### 編集画面

共通化したい要素:

- 親一覧または詳細へ戻るパンくず
- 小見出しは親機能の管理名
- メイン見出しは「新規作成 / 編集 / 登録」などの操作名で統一する
- 説明文は必要最小限にし、フォーム内の補足と役割を分ける
- ヘッダー直下に主要フォームカードを置く

## 基本パターン

### 一覧画面ヘッダー

必須:

- パンくず
- 小見出し
- メイン見出し
- 説明文
- 主要アクションまたは補助リンクの右寄せ領域

推奨:

- ヘッダー本体は 1 枚のカードにまとめる
- 補助リンクはテキストリンク、主要操作はボタンで強弱を分ける
- 件数表示はヘッダー外の summary / filter ブロックへ寄せる

避けたいこと:

- ヘッダー内に件数・filter・flash message・主要ボタンを詰め込みすぎること

### 詳細画面ヘッダー

必須:

- パンくず
- 小見出し
- 対象名の見出し
- 戻るリンクと編集リンク

推奨:

- ヘッダーはカード外でもよいが、左右の情報量を揃える
- ステータスバッジは見出し直下に寄せ、説明文とは役割を分ける

避けたいこと:

- 戻る導線が複数あり、優先順位が読めない状態

### 状態別一覧ヘッダー

必須:

- 親一覧を含むパンくず
- 親機能名の小見出し
- 状態別一覧名の見出し
- 通常一覧とは分けて扱う理由を短く書く説明文
- 通常一覧へ戻るリンク

推奨:

- 追加ボタンを置かず、確認・遷移を主責務にする
- 件数はヘッダーの下に別で置く

### 編集画面ヘッダー

必須:

- パンくず
- 小見出し
- 操作名の見出し

推奨:

- 詳細画面との差を、操作名と補助説明で明確にする
- 主要アクションはフォーム下部に寄せ、ヘッダーは文脈整理に集中させる

## モダンに見せるための最低限の方向性

- 情報の優先順位が一目で分かるよう、見出し・説明文・補助リンクの役割を分ける
- 余白と整列を安定させ、ヘッダー内の左右バランスを揃える
- カード・境界線・見出しサイズの使い方を画面種別ごとに固定する
- 装飾を増やすのではなく、レイアウトルールで整える
- 一覧画面どうし、詳細画面どうしで同じ骨格を使い、機能差は文言とリンクで表現する

## 今後の実装順候補

1. 一覧画面ヘッダーの共通ルールを決める
   - item / outfits / wear logs / purchase candidates の一覧を基準にする
2. 状態別一覧ヘッダーを一覧画面ルールに寄せる
   - invalid / disposed の導線と説明文を揃える
3. 詳細画面ヘッダーの共通ルールを決める
   - 戻るリンク、編集リンク、ステータスバッジ位置を揃える
4. 編集画面ヘッダーの共通ルールを決める
   - 新規作成 / 編集 / 登録の表記とフォーム直下の構造を揃える
5. 必要なら共通 component 化を検討する
   - 先にレイアウトルールを固め、component 化は後段にする

## 保留事項

- 件数表示を一覧ヘッダー内へ寄せるか、summary カードへ分けるかの最終ルール
- モバイル幅で補助リンクと主要アクションをどう折り返すか
- ホーム画面を同じルールにどこまで寄せるか
- settings 配下の画面をこのガイドラインへどの段階で合わせるか
- component 共通化をどこまで行うか
