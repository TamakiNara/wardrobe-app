# Error Message Guidelines

アプリ全体のエラーメッセージ設計の共通ルールを整理する資料。
この資料では、文体、原因の出し方、次にするべき行動の案内、技術詳細の隠し方、重要画面で検討すべき場面を定義する。

---

## 共通ルール

### 文体

- 基本は丁寧語とする
- ユーザーを責める表現は避ける
- 過剰に硬い表現より、落ち着いて対処できるトーンを優先する

### 原因の書き方

- 原因はユーザーが理解できる範囲で簡潔に書く
- 入力エラーは、どの項目を直すべきかが分かる程度まで書く
- サーバー側・通信側の問題は、詳しい原因断定までは書かない

### 次にすること

- 可能な限り、ユーザーが次にとるべき行動を書く
- 「もう一度お試しください」「入力内容をご確認ください」のような行動案内を基本とする
- 再試行で解決しない可能性がある場合は、時間をおいて再試行する案内も検討する

### 技術詳細

- スタックトレース、例外クラス名、SQL、内部 ID などの技術詳細は UI に出さない
- ログや開発者向け情報は別経路で扱う
- ユーザー向けには、影響と対処のみを簡潔に伝える

---

## 重要画面の検討一覧

- ログイン / ユーザー登録
- アイテム登録 / 編集
- 設定保存
- コーディネート登録
- 画像アップロード

---

## 画面別に整理したい論点

### ログイン / ユーザー登録

- 必須項目未入力
- メールアドレス形式エラー
- 認証失敗
- セッション切れ / 通信失敗

### アイテム登録 / 編集

- 必須項目未入力
- 不正な項目値
- 保存失敗
- 一覧 / 詳細から遷移した先での対象データ不在
- 詳細画面での status 変更失敗

### 設定保存

- 保存中の通信失敗
- セッション切れ
- バリデーションエラー

### コーディネート登録

- 構成不足
- 不正なアイテム組み合わせ
- 保存失敗

### 画像アップロード

- ファイル形式エラー
- ファイルサイズ超過
- アップロード失敗

## 初期文言案

### ログイン / ユーザー登録

- 必須項目未入力: 「入力されていない項目があります。内容をご確認ください。」
- メールアドレス形式エラー: 「メールアドレスの形式が正しくありません。」
- 認証失敗: 「メールアドレスまたはパスワードが正しくありません。」
- セッション切れ / 通信失敗: 「通信に失敗しました。時間をおいて再度お試しください。」

### アイテム登録 / 編集

- 必須項目未入力 / 不正な項目値: 「未入力または選択できない項目があります。内容をご確認ください。」
- 保存失敗: 「アイテムを保存できませんでした。時間をおいて再度お試しください。」
- 対象データ不在: 「対象のアイテムが見つかりませんでした。一覧から確認してください。」
- status 変更失敗: 「アイテムの状態を変更できませんでした。時間をおいて再度お試しください。」

### 設定保存

- バリデーションエラー: 「保存できない項目があります。設定内容をご確認ください。」
- 通信失敗: 「設定を保存できませんでした。時間をおいて再度お試しください。」
- セッション切れ: 「ログインが必要です。お手数ですが、もう一度ログインしてください。」

### コーディネート登録

- 構成不足: 「コーディネートに使うアイテムを選択してください。」
- 不正なアイテム組み合わせ: 「選択できないアイテムの組み合わせがあります。内容をご確認ください。」
- 保存失敗: 「コーディネートを保存できませんでした。時間をおいて再度お試しください。」

### 出し分けの初期方針

- 項目単位で直せるものは、項目直下に表示する
- 保存失敗や通信失敗は、画面上部のバナーまたはフォーム直上でまとめて案内する
- ログイン切れは、再ログインへ誘導するバナーを優先し、トースト単体では済ませない

## 長いフォームの送信エラー

縦に長い登録 / 編集フォームでは、field-level error だけに頼ると、送信ボタン付近から見てエラー位置が分かりづらくなる。

### 基本方針

- 送信失敗時は、フォーム上部または送信ボタン付近に error summary を表示する
- error summary では概要文と、修正が必要な項目メッセージ一覧をまとめて案内する
- field-level error は従来どおり各項目の近くに表示する
- 長いフォームでは、error summary と field-level error を併用する

### 可能なら行うこと

- 最初のエラー項目へ自動スクロールする
- 該当 input / select / textarea へ focus する
- error summary に `role="alert"` と `aria-live` を付ける

### current

- 購入検討フォームで先行対応する
- 送信失敗時にフォーム上部へ error summary を出し、最初のエラー項目へスクロールと focus を行う

### planned

- item form
- outfit form
- wear log form
- weather form
- settings form

上記の長いフォームにも、同じ error summary と誘導方針を段階的に適用する

## 今後の決めごと

- 各画面で実際に出すメッセージ文言の確定
- バナー、トースト、項目下エラーの使い分け
- ログイン切れ時のメッセージと遷移導線の統一
- エラーコードを開発者向けにのみ扱うか、UI と連携するかの整理
