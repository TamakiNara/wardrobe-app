# Item Form Structure

## 目的

item form のセクション構成と、各セクションの役割を整理する。  
この資料は item form の構造に関する正本とし、カテゴリ別の補足仕様は `docs/specs/items/*.md` を参照する。

---

## セクション構成

モバイル順を正本とし、現時点の第一候補は以下とする。

1. 基本情報
2. 分類
3. 色
4. 利用条件・特性
5. サイズ・実寸
6. 素材・混率
7. 購入情報
8. 補足情報
9. 画像
10. 送信 actions

### 各セクションの所属項目

#### 基本情報

- 名前
- ブランド

#### 分類

- カテゴリ
- 種類
- 形
- spec

補足:

- `分類` は `カテゴリ / 種類 / 形 / spec` を 1 つの流れとして扱う
- `カテゴリ` はアイテムの大分類であり、最初に選ぶ分類軸として扱う
- `種類` はカテゴリ内の主要分類であり、ユーザーがカテゴリの次に選ぶ軸として扱う
- `形` は保存上は独立した分類軸として扱う
- ただし `形` は常に独立入力にせず、種類から一意に決まる場合は非表示のまま内部で補完してよい
- `種類` だけでは表現しきれない場合にのみ、`形` を補助分類として入力させる
- `spec` は分類の先にある詳細属性として扱い、丈・覆い方・袖・首回り・股上などを含む
- `spec` は独立カードでもよいが、分類の続きとして読める配置を優先する

### `形` の表示ルール

- `形` は保存上は独立軸として維持するが、UI では常に独立入力にしない
- 種類から保存 shape が一意に決まる場合は、`形` は非表示のまま内部補完してよい
- 種類だけでは shape が複数候補に分かれる場合だけ、`形` を補助分類として表示する
- `形` が不要な場合は、グレーアウトではなく非表示とする
- `spec` は `形` の代替ではなく、分類の先にある詳細属性として扱う

## 戻る導線

item new 画面の戻るボタンは、遷移元に応じて文言と遷移先を切り替える。

- 通常の item 新規作成
  - 文言: `一覧に戻る`
  - 遷移先: `/items`
- purchase candidate 由来で `returnTo` がない場合
  - 文言: `購入検討一覧に戻る`
  - 遷移先: `/purchase-candidates`
- purchase candidate 由来で `returnTo` がある場合
  - 文言: `元の画面に戻る`
  - 遷移先: `returnTo`

補足:

- purchase candidate 詳細から item 化した場合は、キャンセル時に purchase candidate 詳細へ戻る
- そのため `returnTo` がある場合は `一覧に戻る` ではなく `元の画面に戻る` と表示する

## `memo` の扱い

- `memo` は自由入力の補足情報とする
- 購入用途に限定しない
- 属性補足・運用メモ・自由記述の受け皿として扱う
- 独立した主要セクションではなく、`補足情報` の中で扱う

---

## 見出し方針

- `購入・補足` 見出しは廃止し、`購入情報` に統一する
- `memo` は `購入情報` には含めず、`補足情報` に配置する
- セクション名は purchase candidate と可能な限りそろえ、用語ブレを避ける

---

## spec の必須/任意方針

spec の必須/任意ルールは、まず item 側の既存実装を起点に定義する。  
最終的には item / purchase candidate で共通化するが、本節では item 側の現行ルールを正本として整理する。

### 基本方針

- spec は「丈・覆い方・袖・首回り・股上などの詳細属性」を扱う
- 必須化するのは、未入力だと分類や比較に必要な粒度が欠ける項目に限る
- default 値で自然に補完できる項目や、補助的な差分項目は任意とする

### item 側の現行ルール

#### tops

- `sleeve`: 任意
- `length`: 任意
- `neck`: 任意
- `design`: 任意
- `fit`: 任意

理由:

- tops は shape ごとに候補集合と default があり、未入力でも自然に補完できる項目が多い
- 現行 item form でも未選択エラーは出さない

#### bottoms / pants

- `length_type`: 必須
- `rise_type`: 任意

理由:

- `length_type` は丈の分類軸として比較・識別に必要
- `rise_type` は補助属性として扱う

#### skirts

- `length_type`: 必須
- `material_type`: 任意
- `design_type`: 任意

理由:

- `length_type` は丈の分類軸として比較・識別に必要
- `material_type` と `design_type` は補助属性として扱う

#### legwear

- `coverage_type`: 条件付き必須
- `socks` / `leggings` では必須
- それ以外では非表示または任意とする

理由:

- socks / leggings では coverage が覆い方の主分類軸になるため
- それ以外の legwear では必須項目として扱わない

### 補足ルール

- `shape` は `spec` には含めず、分類の別要素として扱う
- `shape` は保存上は独立軸として維持するが、UI では常に独立入力の主役にしない
- `shape` の入力欄が不要な場合は、グレーアウトではなく非表示とする
- `shape` は分類軸の正本として扱う
- tops では旧データに `spec.tops.shape` 相当の値が残りうるが、これは恒久仕様ではなく旧互換値としてのみ扱う
- 現行の request / restore / submit / response は `shape` を分類軸の正本として扱い、`spec.tops.shape` を正規ルートとしては参照しない
- backend に `spec.tops.shape` が送られてきても、保存値や response の正本としては扱わない
- 古いローカル下書きが残っている場合は、必要に応じて手動クリアを前提とする
- item 側は既存互換のため、当面 `形` UI が残るカテゴリがある
- `spec` の必須/任意ルールは、UI 表示だけでなく validation や restore 時の挙動とも整合するように扱う
- purchase candidate 側への適用方針は `docs/specs/purchase-candidates.md` 側で管理する

## 補足

- この資料はフォーム構造の正本であり、PC 幅での段組みは別途検討する
- `補足情報` は、将来のタグ・履歴・内部メモなどを追加する受け皿としても使える設計を第一候補とする
- タグ機能は将来追加予定だが、現時点では未実装とし、方針メモは `docs/specs/tags.md` を参照する

## 固定実寸

- 固定実寸の正本は `web/src/lib/items/size-details.ts`
- item form は `category + shape` から固定実寸を解決する
- 対応がない場合は自由項目の実寸で補う
- 実寸値は `value / min / max / note` を持てる
- `note` は `ヌード寸` / `約` / `後ろ約` などの補足注記で、表示時は値の前に付ける
- `total_length` は現状 `総丈` として表示する
- `スカート丈` は `総丈` と意味が異なるため、別項目化する場合は要再判断とする
- 対応範囲の一覧は [size-details.md](./size-details.md) を参照
