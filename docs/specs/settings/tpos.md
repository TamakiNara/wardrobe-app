# TPO 任意追加仕様

## 目的

本仕様は、Wardrobe App における **TPO（利用シーン）をユーザーごとの管理対象として扱うための仕様** を定義する。  
現状は `仕事 / 休日 / フォーマル` を前提にした固定的な運用をしているが、今後はユーザーごとに必要な TPO を追加・無効化・並び替えできるようにし、item / outfit / wear log の各機能で一貫して利用できるようにする。

---

## 概要

TPO 任意追加は、設定配下で管理する **ユーザー単位の TPO マスタ機能** とする。  
TPO は自由入力で都度保存するのではなく、**あらかじめ登録済みの TPO から選択する方式** を前提とする。  
これにより、表記ゆれや重複登録を抑えつつ、一覧・絞り込み・将来拡張で扱いやすい状態を保つ。

---

## 背景

現在のアプリでは、TPO は実質的に固定値に近い扱いで運用している。  
一方で、実際の利用シーンはユーザーごとに差があり、以下のような要望に対応しにくい。

- `通院`
- `旅行`
- `推し活`
- `雨の日`
- `在宅勤務`
- `セミフォーマル`

TPO をユーザーごとに管理可能にすることで、既存の `仕事 / 休日 / フォーマル` をベースにしつつ、必要な利用シーンを追加できるようにする。  
ただし、TPO はタグのような完全自由分類ではなく、**選択肢として制御されたマスタ** として扱う。

---

## 位置づけ

- 設定系機能
- ユーザー単位マスタ
- item / outfit / wear log から参照される共通選択肢
- タグ機能の代替ではない
- 将来の絞り込み・集計・候補提案に使いやすい基盤

## 現状の実装 / 今後対応 / 要再判断

### 現状の実装

- TPO は複数選択の概念として、item / outfit / wear log で使う前提がある
- settings + item + outfit では、TPO の選択肢正本を `user_tpos` として扱う
- settings + item + outfit の初期版保存方式は ID ベースとする
- プリセットとして `仕事 / 休日 / フォーマル` を初期投入する
- ユーザー作成時点で、`user_tpos` にプリセット TPO を作成する
- inactive TPO は新規候補に出さず、既存データに含まれる場合は表示・保持する
- プリセット TPO の名称変更は不可
- 並び替え UI は初期版では上下移動を採用する
- 現状の API / 画面として `/settings/tpos` `GET /api/settings/tpos` `POST /api/settings/tpos` `PATCH /api/settings/tpos/{id}` を持つ

### 今後対応

- wear log で同じ TPO 正本を使う段階対応
- wear log の保存方式を ID ベースへ揃えること
- inactive TPO の表示ルールを wear log の一覧 / 詳細 / 編集へ展開すること

### 要再判断

- wear log で TPO をどの粒度まで使うか
- wear log で inactive TPO をどの画面まで表示するか
- item / outfit の `tpos` 互換フィールドを、どの段階で API / DB から外すか

## 基本方針

### 1. TPO はユーザーごとのマスタで持つ

TPO は全ユーザー共通マスタではなく、**ユーザー単位の管理データ** とする。  
理由は、必要な利用シーンがユーザーごとに異なるためである。  
ただし、初期状態で何も選べないと使いにくいため、プリセット TPO を各ユーザーへ初期投入する。

### 2. 自由入力ではなく登録済み TPO から選ぶ

item / outfit / wear log 側では、TPO を自由入力で都度保存しない。  
**登録済み TPO の選択式** とし、表記ゆれ・重複・ typo を防ぐ。

初期版では、settings / item / outfit は **TPO ID を正本として扱う**。  
wear log は後段対応とし、初期版の対象外に置く。

### 3. 削除より無効化を優先する

TPO は既存データとの整合のため、物理削除より **無効化** を優先する。  
無効化した TPO は、新規入力では選択肢から外すが、既存データでは保持・表示できる状態を前提とする。

### 4. 並び順はユーザーが管理できる

プリセット TPO とユーザー追加 TPO を分けて固定順にするのではなく、**同じ並び替え対象** として扱う。  
表示順は `sort_order asc` を正本とする。

## TPO の役割

TPO は「その item / outfit / wear log がどの利用シーンに向くか、または使われたか」を示す補助分類とする。

### item

- そのアイテムが向いている利用シーン
- 例: `仕事`, `休日`, `通院`

### outfit

- そのコーディネートが向いている利用シーン
- 例: `フォーマル`, `旅行`

### wear log

- 実際にその日にどの利用シーンで使ったか
- 将来的に item / outfit と異なる文脈を持つ可能性はあるが、当面は同じ TPO 選択肢を使う

---

## プリセット TPO

### 初期プリセット候補

- `仕事`
- `休日`
- `フォーマル`

### 扱い

プリセット TPO は、各ユーザーに初期投入される標準選択肢とする。
現時点の実装では、ユーザー作成時に `user_tpos` へ作成する。  
既存ユーザーは `user_tpos` 導入 migration で backfill し、runtime の補完処理は防御コードとして残す。  
役割分担としては、既存ユーザーは migration で一括補完し、新規ユーザーは作成時に初期投入し、想定外の経路で `user_tpos` が欠けていた場合だけ runtime 補完で救済する。

#### 属性

- `is_preset = true`
- 削除不可
- 無効化可
- 名称変更不可

### 理由

プリセットの名称変更を許可すると、画面上は柔軟だが、仕様共有やサポート時に「同じ意味の TPO が人ごとに別名になる」問題が起きやすい。  
一方で、不要な場合は無効化できれば十分なケースが多いため、初期方針としては **名称変更不可・無効化可** を推奨する。

---

## ユーザー追加 TPO

ユーザーは、自分に必要な TPO を追加登録できる。

### 追加条件

- 同一ユーザー内で重複不可
- 前後空白除去
- 空文字不可
- 文字数上限あり

### 想定例

- `通院`
- `旅行`
- `在宅勤務`
- `推し活`
- `雨の日`

### 補足

ユーザー追加 TPO も、プリセットと同様に無効化対象とする。  
ただし、プリセットと異なり、名称は作成者自身が決めるため、更新可否は将来拡張余地がある。

---

## 削除と無効化

### 方針

- 削除より無効化を優先する
- 非 active TPO は新規選択肢に出さない
- 既存データでは保持し、表示できるようにする
- プリセット TPO も追加 TPO も無効化対象に含める

### 推奨運用

#### プリセット TPO

- 物理削除しない
- 無効化のみ許可

#### ユーザー追加 TPO

- 初期実装範囲では物理削除を持たず、無効化で対応する
- 将来、本当に未使用であれば削除可能にする案はありうるが、初期版では不要

### 理由

TPO は item / outfit / wear log に紐づくため、削除時に参照整合や表示ルールが複雑になる。  
そのため、まずは **無効化で管理する方が安全** である。

---

## 並び順

### 方針

- TPO はユーザーが並び順変更可
- プリセット TPO も追加 TPO も同じ並び替え対象に含める
- 管理用に `sort_order` を持つ
- 一覧表示は `sort_order asc` を正本とする

### UI 上の扱い

初期版では、複雑な drag and drop よりも、上下移動ボタンの方が実装・テスト・保守の観点で扱いやすい。  
初期版では **上下移動ボタン** を採用する。

---

## データ構造案

### `user_tpos`

| column | type | description |
| --- | --- | --- |
| id | bigint | 主キー |
| user_id | bigint | 所有ユーザーID |
| name | string | TPO 名 |
| sort_order | unsigned int | 表示順 |
| is_active | boolean | 有効フラグ |
| is_preset | boolean | プリセット由来フラグ |
| created_at | timestamp | 作成日時 |
| updated_at | timestamp | 更新日時 |

### 制約案

- `unique(user_id, normalized_name)` 相当の重複防止
- `sort_order` はユーザー内で一意に近い運用を想定
- `user_id` は `users.id` を参照

### 補足

重複判定では、少なくとも前後空白除去と大文字小文字・全半角揺れを吸収できる余地を残す。  
正式な正規化ルールは実装時に別途定義してよい。

## API

### `GET /api/settings/tpos`

#### 目的

- 現在のユーザーが持つ TPO 一覧取得
- settings 管理画面表示
- settings 管理画面表示
- item / outfit の選択肢取得

#### response 例

```json
{
  "tpos": [
    {
      "id": 1,
      "name": "仕事",
      "sortOrder": 1,
      "isActive": true,
      "isPreset": true
    },
    {
      "id": 2,
      "name": "休日",
      "sortOrder": 2,
      "isActive": true,
      "isPreset": true
    },
    {
      "id": 3,
      "name": "通院",
      "sortOrder": 4,
      "isActive": false,
      "isPreset": false
    }
  ]
}
```

### `POST /api/settings/tpos`

#### 目的

- ユーザー追加 TPO の新規作成

#### request 例

```json
{
  "name": "通院"
}
```

#### 方針

- 作成時の `sort_order` は末尾自動採番
- `is_active = true`
- `is_preset = false`

### `PATCH /api/settings/tpos/{id}`

#### 目的

- 名称更新
- 有効 / 無効切替
- 並び順変更

#### request 例

```json
{
  "name": "旅行",
  "isActive": true,
  "sortOrder": 5
}
```

#### 補足

初期版では「名称更新」と「並び順変更」を同一 endpoint で扱ってもよい。  
ただし、並び順変更を複数件一括更新にしたくなった場合は、将来 `reorder` 専用 API を切る余地がある。

## item / outfit / wear log との関係

### 基本方針

TPO マスタは settings 側に置くが、**利用先は item / outfit / wear log の共通選択肢** とする。  
Phase 1 では settings + item + outfit を今回の対象とし、wear log は後続で対応する。

### item

- 登録済み TPO から複数選択
- 初期版では TPO ID を保存する
- 非 active TPO は新規選択肢に出さない
- ただし、既存 item がすでに持っている非 active TPO は表示・保持可能にする

### outfit

- item と同様
- 初期版では TPO ID を保存する
- 既存 outfit に紐づいている非 active TPO は表示・保持可能にする

### wear log

- 初期版の対象外とし、後段で対応する
- item / outfit と同じ選択肢を使う方向
- ただし、wear log で TPO をどこまで必須にするかは未確定

## 既存保存構造との関係

これは Phase 1 実装時に整理しておくべき論点である。

### 現時点の課題

既存の item / outfit / wear log は、TPO を配列値として扱っている前提が強い。  
ここに `user_tpos` を導入する場合、初期版では item / outfit を ID ベースへ寄せ、wear log は後段で扱う。

### 初期版の方針

- settings 側の選択肢正本を `user_tpos` に置く
- item / outfit の保存値は TPO ID ベースとする
- wear log は後段対応とし、初期版では保存構造を変更しない

#### 利点

- 名称変更不可・無効化・並び順管理と整合しやすい
- settings 側の正本と利用先の参照先を揃えやすい
- 将来の集計や絞り込みへ拡張しやすい

#### 注意点

- item / outfit 側の既存保存構造と API / UI の更新が必要になる
- 初期版で wear log を除外する前提を docs 上で明確に分ける必要がある

## バリデーション方針

### 追加

- `name` 必須
- 前後空白除去後に空文字不可
- 文字数上限あり
- 同一ユーザー内で重複不可
- 無効化済み名称との重複も不可とする

### 更新

- 対象は自分の TPO のみ
- プリセット TPO の名称変更は不可
- `is_active` は boolean
- `sort_order` は 1 以上の整数

---

## テスト観点

### 一覧

- 自分の TPO のみ返る
- `sort_order asc` で返る
- プリセットと追加 TPO が混在して返る
- 無効化済みも管理画面用途では返せる

### 追加

- TPO を追加できる
- 同一ユーザー内重複はエラー
- 前後空白除去後の空文字はエラー
- 作成時は末尾 `sort_order` になる

### 更新

- 無効化できる
- 再有効化できる
- 並び順変更できる
- 他人データは更新不可
- プリセット名称変更不可のテストが必要

### 利用先

- item / outfit の選択肢として active TPO のみ出せる
- 既存データに含まれる inactive TPO は表示・保持できる
- wear log は初期版対象外であることを明示できる

---

## 実装時の注意

- TPO をタグのような自由入力に戻さない
- settings 側だけで閉じず、利用先の表示責務まで意識する
- 無効化した TPO を既存データから一律削除しない
- 並び順ロジックを frontend に寄せすぎない
- wear log は初期版対象外であることを docs / 実装依頼で明示する

---

## この仕様で固定する前提

1. TPO の選択肢正本を `user_tpos` にする
2. 初期版の保存方式は ID ベースにする
3. プリセット TPO は `仕事 / 休日 / フォーマル`
4. 削除より無効化を優先する
5. 並び順は `sort_order` で持つ
6. settings 配下に独立画面を置く
7. 初期版の対象は settings + item + outfit とし、wear log は後段に分ける

## 現時点のまとめ

TPO 任意追加は、ユーザーごとの利用シーンを柔軟に扱うための **設定系マスタ機能** として導入する。  
初期版では、

- プリセット TPO を持つ
- settings + item + outfit を対象にする
- 保存方式は ID ベースとする
- inactive TPO は新規候補に出さず、既存データに含まれる場合は表示・保持する
- wear log は後段対応として切り分ける

という前提で整理する。  
一方で、wear log への適用粒度や inactive TPO の詳細表示方法は、引き続き `要再判断` に残す。
