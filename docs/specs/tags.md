## タグ機能

### 基本方針

- タグは分類ではなく補助情報として扱う
- category / subcategory / shape / spec の代替には使わない
- item / outfit / purchase candidate で共通に使える設計を第一候補とする

### 主な用途

- 状態
  - 例: `サイズアウト` / `要補修` / `お気に入り`
- 使用シーン補助
  - 例: `部屋着` / `旅行用`
- コーデ用途
  - 例: `着回しやすい` / `主役` / `インナー用`
- 管理メモ
  - 例: `出品予定` / `情報未整理`
- 主観的評価
  - 例: `似合う` / `着痩せ`

### 設計方針

- ユーザー単位のタグ
- 自由入力
- 複数付与可能
- item / outfit / purchase candidate 共通で使える想定
- 将来的に検索・フィルタ対象にする

### 今回の扱い

- 実装は未着手
- まずは仕様メモとしてのみ整理する
- DB / API / UI の詳細設計は後続で決める

---

## セットアップ可

### 現状方針

- `セットアップ可` はタグで管理する
- 例:
  - `セットアップ可`

### 理由

- 軽量に始められる
- 関係データを持たずに運用できる
- 実データを入れながら、どこまで厳密な関係が必要か見極められる

### 補足

- この段階では「セットアップである可能性」を示すだけで、どの item と対応するかまでは表せない
- その限界を超えたくなった時点で、将来のセットアップリンク機能を検討する

---

## セットアップリンク機能

### 背景

- ジャケットとパンツなど、明確に対応する item 同士を紐づけたいケースがある
- タグだけでは「どの item とセットか」が分からない

### 将来案

item 間の関係データを持つ。

例:

```text
item_relations
- id
- user_id
- source_item_id
- target_item_id
- relation_type
- memo
```

### relation_type の候補

- `setup_pair`
- `color_variant`
- `similar`
- `replacement`

### 想定する UI イメージ

- item 詳細に `関連アイテム` セクションを出す
- セットアップ対象の一覧表示を出す
- 将来的にはコーデ作成時に関連 item を候補として出す

### 今回の扱い

- 実装しない
- 当面はタグで代替する
- 将来拡張候補として docs に残す

---

## 実装順の第一候補

1. item / outfit / purchase candidate 共通のタグ方針を固める
2. user 単位のタグ保存と複数付与を最小実装する
3. 検索 / フィルタ対応を追加する
4. 実データ運用で不足が見えたら、セットアップリンクの独立実装を検討する
