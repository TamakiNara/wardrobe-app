# 次期機能検討メモ

このメモは、未実装または未完成の機能候補を優先順位付きで整理するための planning 用メモです。実装済み機能の詳説は避け、今後どこを詰めるか、何を保留にするかを短く共有することを目的とします。
current 実装の補足や短期引き継ぎではなく、「次に何を決めるか」を残すための planning メモとして使います。current 実装メモや最近の変更履歴は `docs/project/implementation-notes.md` を参照します。

## 優先順位

1. item状態管理の完成
2. 素材・混率管理
3. 検索・絞り込み強化
4. 分析・可視化
5. docs の整合更新

## 1. item状態管理の完成

- 結論: `disposed` を item の基本運用状態として完成させ、delete は登録ミスや重複修正用に限定する。
- 目的 / 理由: 手放し済み item を履歴として残しつつ、outfit / wear log / 候補生成との整合を取りやすくするため。
- 今決めるべきこと: `disposed` / `reactivate` の専用操作導線、`disposed` item 一覧の要否、related outfit を invalid 化する条件、reactivate 時に outfit を自動 restore しないこと、wear log / outfit 候補からの除外ルール。
- 画面への影響: item 詳細・一覧・編集で状態表示と警告導線を揃える。必要なら `disposed` 一覧または filter を追加する。
- API への影響: `status` は通常 update payload に含めず、専用 action で `dispose` / `reactivate` を扱う前提を維持する。
- DB への影響: 当面は `status` 中心で整理し、`disposed_at` や `dispose_reason` は保留とする。
- テスト観点: dispose / reactivate の認可、related outfit invalid 化、候補除外、一覧表示状態、delete との役割分担。
- 今は保留でよいこと: `disposed_at`、`dispose_reason`、状態変更履歴の詳細ログ構造。

## 2. 素材・混率管理

- 結論: item の素材は文字列1本ではなく、複数明細 + 混率で保持する前提で進める。
- 目的 / 理由: 後続の表示・検索・可視化で再利用しやすくし、素材グラフ化の基盤を先に揃えるため。
- 今決めるべきこと: 明細構造の最小 schema、合計 100% バリデーション、未入力許容、主要素材のデフォルト選択肢と自由入力の両立。
- 画面への影響: item create / edit に複数行入力 UI を追加し、素材名と混率を行単位で編集できるようにする。
- API への影響: item upsert payload と response に素材明細配列を追加し、入力がある場合のみ 100% 検証を行う。
- DB への影響: 明細テーブルまたは JSON 明細で、素材名と混率を複数件保持できる構造が必要。
- テスト観点: 合計 100% の正常 / 異常、明細追加削除、主要素材選択 + 自由入力、未入力許容。
- 今は保留でよいこと: 素材マスタの全面運用、同義語吸収、高度な将来拡張。

## 3. 検索・絞り込み強化

- 結論: item 一覧を起点に、状態・雨対応・ブランド・色・素材の filter を段階追加する。
- 目的 / 理由: 基本検索だけでは探し直しが弱く、日常運用で「今見たい item」にすぐ辿れないため。
- 今決めるべきこと: UI で常設する条件、URL クエリ命名、複数条件の組み合わせ方、空状態メッセージ。
- 画面への影響: item 一覧 filter UI の拡張が中心。必要に応じて active / disposed の見え方も調整する。
- API への影響: query builder に寄せやすい単純条件を優先し、一覧 query へ filter パラメータを追加する。
- DB への影響: ブランドや色は現行構造で対応可能だが、素材は明細化後の filter 前提で整理する。
- テスト観点: 条件単体と複合条件、URL 共有、空結果、hidden category との組み合わせ。
- 今は保留でよいこと: outfits / wear logs 側の同等 filter 拡張、検索サジェスト、全文検索。

## 4. 分析・可視化

- 結論: まずはダッシュボード的な簡易集計に留め、集計ロジックの正本を先に固める。
- 目的 / 理由: 利用頻度や未着用状況を把握し、手放し判断や購入判断に繋げるため。
- 今決めるべきこと: worn のみを集計対象にするか、disposed item を集計に含めるか、planned / worn 件数比較の定義。
- 画面への影響: ホームまたは分析画面に、直近30日の着用回数、未着用期間、最終着用日、よく使うカテゴリ / 色を簡易表示する。
- API への影響: 集計用 endpoint または summary payload が必要になるが、まずは単純集計で十分。
- DB への影響: 現時点では wear logs と item status を正本に集計可能で、専用集計テーブルやバッチは不要。
- テスト観点: worn / planned の切り分け、disposed item の扱い、日付境界、カテゴリ / 色集計の母数。
- 今は保留でよいこと: 高度なグラフ UI、バッチ集計、長期予測。

## 5. docs の整合更新

- 結論: 実装済み / 未実装 / planned の境界を docs 上で明確にし、検討対象を独立メモとして維持する。
- 目的 / 理由: 実装認識のずれを減らし、引き継ぎ時に「次に何を決めるか」がすぐ分かる状態を保つため。
- 今決めるべきこと: どの docs を正本にするか、planning メモと実装メモの役割分担、相互リンクの最小セット。
- 画面への影響: 直接の UI 変更はないが、画面仕様の書き分けを current / planned で明示する。
- API への影響: 仕様変更を伴う場合だけ OpenAPI を追随し、検討段階のものは planning docs に留める。
- DB への影響: 新規機能候補の保存方針だけを短く残し、未確定事項は先に固定しない。
- テスト観点: docs 対応時は、実装変更を伴う項目だけ関連 test / seed の要否を合わせて確認する。
- 今は保留でよいこと: README の全面改稿、大規模棚卸し、実装済み機能の再要約。
