# 日本語ファイル編集 Skill

## 目的

日本語を含むファイルの編集時に、文字化け、表示崩れ、重複、欠落、意図しない大規模差分を防ぐ。
特に PowerShell 経由の文字化け事故、長文 docs の広範囲破損、壊れた版を起点に修正を重ねるループを防ぐ。

## 適用対象

- 日本語コメントを含むソースコード
- 日本語 docs、README、仕様書、議事メモ
- 日本語を含む `.test.ts` / `.test.tsx`
- 長文の Markdown / テキストファイル

## 編集手段の優先順位

1. `apply_patch` による最小差分編集
2. Python スクリプトによる編集（`encoding="utf-8"` を明示）
3. それ以外の手段

## 禁止事項

- PowerShell の `Set-Content` / `Add-Content` / `Out-File` による日本語ファイル編集
- シェルの置換コマンドで日本語文字列を直接埋め込むこと
- 日本語を含むファイル全体を再生成して置き換えること（明示指示がある場合を除く）
- `�` を見つけた箇所だけを見て推測で埋めること
- PowerShell の出力を正として上書き保存すること
- 壊れた版を元にさらに文言修正を重ねること
- 同じファイルを複数ツールで開いたまま編集すること

## 基本手順

1. 編集前に文字コードと改行コードを確認する
2. 対象見出し、対象段落、対象行付近を特定する
3. 最小差分で編集する
4. UTF-8 として再読込する
5. 修正箇所だけでなく、前後 20〜50 行と見出し末尾まで確認する
6. `git diff --word-diff` または通常 diff で差分を確認する
7. `?`、`�`、文字化け、見出し重複、箇条書き重複がないか確認する
8. 必要なら `rg "�|\?" <file>` 等で異常文字や `?` を確認する

## 文字化け・表示崩れを見つけた場合

`�`、`??`、`????`、不自然な記号列、想定外の空文字化を見つけた場合は、その場で置換を続けず、まず停止する。

切り分け手順:

1. PowerShell やターミナル表示だけで文字化けと断定しない
2. UTF-8 で再読込する
3. Unicode code point や escaped form でも実データを確認する
4. 表示上の問題か、保存済みファイル自体の破損かを切り分ける

対応方針:

- 表示だけが崩れている場合は、その表示を根拠に上書きしない
- 保存済みファイル自体が壊れている場合は、その壊れた状態を直接修正し続けない
- Git 上の直前正常版または退避した原本を基準に、必要な変更だけを再適用する
- 文字化け修正と文言修正を同じ差分で同時に進めない。可能なら分ける
- 文字コード起因の不整合が少しでも疑われる場合は、追加修正より先にロールバックまたは正常版からの再適用を優先する

## 長文日本語ファイルの編集方針

長文の日本語ファイル（docs、仕様書、議事メモ、README など）は、原則として全文再生成・全文置換を禁止する。
必ず、対象見出し・対象段落・対象行付近のみの最小差分で編集すること。
1つの指示で複数見出しを同時に大きく編集しないこと。
必要な場合は見出し単位で分割して編集すること。
文字化けや表示崩れが疑われる状態で、長文ファイルに対して追加の文言修正を重ねないこと。

## 長文日本語 docs への新規追記手順

長文日本語 docs に新しい日本語行・段落・箇条書きを追加する場合は、直接その日本語本文を挿入しないこと。
原則として、以下の二段階で追記すること。

1. `apply_patch` で、追記位置に ASCII のみからなる一意なプレースホルダを最小差分で挿入する
2. Python スクリプトで、そのプレースホルダだけを UTF-8 の日本語本文に置換する

追加ルール:

- Python では `encoding="utf-8"` を明示し、改行コード維持のため read/write ともに `newline=""` を使う
- Python に渡す新規日本語本文は、PowerShell 上に生の日本語文字列として直接書かず、Unicode escape・code point・UTF-8 byte 列などから Python 側で復元する
- 置換対象は一意な ASCII プレースホルダ 1 箇所に限定し、前後の既存日本語本文をキーにした広い置換を避ける
- 追記後は、追加した見出しまたは節の末尾まで再読込し、重複・欠落・`?` 混入がないことを確認する

## 日本語文字列を含む置換の原則

日本語文字列をキーにした一括置換は、対象が一意に特定できることを確認してから行うこと。
同一文字列が複数存在する可能性がある場合は、見出し、前後文脈、開始位置と終了位置を確認し、狭い範囲に限定して編集すること。
文字化けが疑われる状態では、日本語文字列そのものをキーにした広い置換を避けること。
行番号ベースの一括置換は最終手段とし、実行した場合は直後に対象ブロック全体を再読込して、重複行・欠落行・見出し重複がないか必ず確認する。

## ファイル別の追加確認

- 日本語を含む `.test.ts` / `.test.tsx` を複数編集した場合は、変更対象の test ファイル全体を UTF-8 で再読込みし、実データとして `?` / `????` が残っていないか追加確認する
- `docs/data/database.md` や `docs/project/implementation-notes.md` のような長文 docs を更新した場合は、変更後に「`?` を含む行が残っていないか」と「同じ見出しや箇条書きが重複していないか」を追加で確認する
- 同じ依頼で日本語を含むファイルを複数編集した場合は、変更対象の全ファイルを UTF-8 で再読込し、修正した文字列そのものが意図どおり入っていることを確認し終えるまで完了報告しない

## 異常を検知した場合

編集後に文字化け、見出し重複、断片の欠落、想定外の大量差分を検知した場合は、そのまま編集を続けず停止すること。

報告内容:

- どのファイルで
- どの操作後に
- 何が起きたか

追加方針:

- `�`、`??`、`????`、不自然な記号列、想定外の空文字化を見つけた場合は、その場で埋めずに停止する
- 停止後は、現在ファイルを直接直し続けるかではなく、正常版へ戻すか、正常版を基準に再適用するかを先に判断する
- 差分上の一部だけが直っていても、ファイル全体確認が終わるまでは復旧完了と判断しない

## 完了条件

以下をすべて満たした場合のみ完了とする。

- 修正対象ファイルを UTF-8 で再読込し、修正箇所とその前後に `�`、`??`、`????`、欠落文字、不自然な記号列がない
- `git diff` 上で意図しない大規模差分が出ていない
- 見出し単位または節単位で、差し替え範囲の開始位置と終了位置が崩れていない
- 箇条書き、コードブロック、バッククォート、見出しレベルが壊れていない
- 文字化け修正の場合、差分確認とファイル実体確認の両方が終わっている
- 同じ依頼で複数ファイルを編集した場合、対象全ファイルの再読込確認が終わっている
