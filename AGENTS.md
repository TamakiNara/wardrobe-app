# 開発ルール

## プロジェクト前提

- GitHub の対象ブランチ / 対象ファイルを正本とする
- docs / OpenAPI / 実装 / test の整合を重視する
- 曖昧なものは `current / planned / 要再判断` を分けて扱う
- 変更範囲に関係ない大規模修正は行わない

## 文字コード・言語・改行コード

- 常に UTF-8 エンコードでファイルを読み書きする
- PowerShell やターミナルの表示だけで文字化けと断定しない
- 日本語を escaped Unicode に変換しない
- 明示的な指示がない限り、日本語コメント / 日本語 docs / 日本語メモを勝手に英語化しない
- 各ファイルの既存の改行コードを維持する
- ファイルが CRLF の場合は CRLF を維持する
- ファイルが LF の場合は LF を維持する

## 承認時の質問文

- 権限昇格や承認が必要な操作を行う際の質問文は、日本語で記載する

## コミットメッセージ提案時の確認

ユーザーがコミットメッセージを求めたタイミングでは、コミット対象の変更に応じて以下もあわせて確認し、必要なら先に反映してからコミットメッセージを提案する。

- 関連するテストの実装・修正
- `docs/` 配下の仕様書・実装メモ・設計資料の追加・修正

原則:

- 実装変更がある場合は、対応するテストが不足していないか確認する
- 挙動や仕様が変わる場合は、`docs/` に正本またはメモとして反映する
- コミットメッセージは、テストと docs の反映まで含めた実際の変更内容に合わせて提案する
- コミットメッセージは英語で出力する
- ユーザーが明示的に求めていなくても、作業完了を報告する際は差分があるときのみコミットメッセージ候補もあわせて出力する（調査のみで差分がない場合は不要）

## 確認結果の報告

- テスト、lint、format check などの確認自体は継続して行う
- 作業完了時の報告は毎回「確認済みです」の一文だけを出せばよい
- テストや format / lint の確認を実施できなかった場合、またはユーザーに手動確認を依頼したい場合のみ、その旨を実施してほしいコマンドとあわせて明示する

## 使用する Skill

状況に応じて以下の Skill を参照すること。

- 日本語を含むファイルの編集、長文日本語 docs への追記、文字化け疑いの切り分け、復旧、完了条件:
  - `.agents/skills/japanese-file-editing/SKILL.md`
- commit 前の lint / prettier / pint / test / hook 確認:
  - `.agents/skills/pre-commit-checks/SKILL.md`
