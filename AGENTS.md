# 開発ルール

## プロジェクト前提

- GitHub の対象ブランチ / 対象ファイルを正本とする
- docs / OpenAPI / 実装 / test の整合を重視する
- 曖昧なものは `current / planned / 要再判断` を分けて扱う
- 変更範囲に関係ない大規模修正は行わない

## 文字コード

常に UTF-8 エンコードでファイルを読み書きすること。
文字化けが見つかった場合も、そのファイルは UTF-8 である前提で UTF-8 として再読み込みすること。

## 言語

ソースコード中のコメントには日本語が含まれる場合がある。
日本語を escaped Unicode に変換してはいけない。
明示的な指示がない限り、日本語コメントを変更しないこと。
日本語で記載されている docs やメモ、日本語を扱うべきファイルは、明示的な指示がない限り勝手に英語化しないこと。
一時的な逃げとして英語へ置き換えることも禁止する。

## 改行コード

各ファイルの既存の改行コードを維持すること。
ファイルが CRLF の場合は CRLF を維持する。
ファイルが LF の場合は LF を維持する。
明示的な指示がある場合、またはリポジトリ設定で必要な場合を除き、改行コードを変更しないこと。

## 承認時の質問文

権限昇格や承認が必要な操作を行う際の質問文は、日本語で記載すること。

## コミットメッセージ提案時の確認

ユーザーがコミットメッセージを求めたタイミングでは、コミット対象の変更に応じて以下もあわせて確認し、必要なら先に反映してからコミットメッセージを提案する。

- 関連するテストの実装・修正
- `docs/` 配下の仕様書・実装メモ・設計資料の追加・修正

原則:

- 実装変更がある場合は、対応するテストが不足していないか確認する
- 挙動や仕様が変わる場合は、`docs/` に正本またはメモとして反映する
- コミットメッセージは、テストと docs の反映まで含めた実際の変更内容に合わせて提案する
- コミットメッセージは英語で出力する
- ユーザーが明示的に求めていなくても、作業完了を報告する際はコミットメッセージ候補もあわせて出力する

## 日本語を含むファイルの編集手順

日本語を含むファイルは、以下の優先順位で編集すること。

1. `apply_patch` による最小差分編集
2. Python スクリプトによる編集（`encoding="utf-8"` を明示）
3. それ以外の手段

禁止事項:

- PowerShell の `Set-Content` / `Add-Content` / `Out-File` による日本語ファイル編集
- シェルの置換コマンドで日本語文字列を直接埋め込むこと
- 日本語を含むファイル全体を再生成して置き換えること（明示指示がある場合を除く）

必須確認:

- 編集前に文字コードと改行コードを確認する
- 編集後に対象箇所を再読込して、日本語・改行・前後文脈が崩れていないか確認する
- PowerShell 経由で日本語が `?` に置き換わって保存される事故に注意する
- PowerShell の表示が `????` に見えても、すぐに文字化けと断定しない。まず UTF-8 で再読込し、Unicode code point や escaped form で実データを確認する
- PowerShell 経由で日本語を新規書き込みする必要がある場合は、Python 側で Unicode escape から復元して UTF-8 で書き出す方法を優先する
- `git diff` で対象外の大量差分が出ていないか確認する
- `docs/` 配下で見出し単位の修正をする場合は、置換前に開始見出しと終了見出しを確認し、差し替え範囲を明確にしてから編集する
- 行番号ベースの一括置換は最終手段とし、実行した場合は直後に対象ブロック全体を再読込して、重複行・欠落行・見出し重複がないか必ず確認する
- `?` を含む行を修正した後は、その周辺数行だけでなく、同じ節の末尾まで確認し、壊れた断片や重複が残っていないか確認する
- `docs/data/database.md` や `docs/project/implementation-notes.md` のような長文 docs を更新した場合は、変更後に「`?` を含む行が残っていないか」と「同じ見出しや箇条書きが重複していないか」を追加で確認する

## 長文日本語ファイルの編集方針

長文の日本語ファイル（docs、仕様書、議事メモ、README など）は、原則として全文再生成・全文置換を禁止する。
必ず、対象見出し・対象段落・対象行付近のみの最小差分で編集すること。
1つの指示で複数見出しを同時に大きく編集しないこと。
必要な場合は見出し単位で分割して編集すること。

## 長文日本語 docs への新規追記手順

`docs/specs/error-message-guidelines.md` のような長文日本語 docs に対して、新しい日本語行・段落・箇条書きを追加する場合は、直接その日本語本文を挿入しないこと。
原則として、以下の二段階で追記すること。

1. `apply_patch` で、追記位置に ASCII のみからなる一意なプレースホルダを最小差分で挿入する
2. Python スクリプトで、そのプレースホルダだけを UTF-8 の日本語本文に置換する

追加ルール:

- Python では `encoding="utf-8"` を明示し、改行コード維持のため read/write ともに `newline=""` を使うこと
- Python に渡す新規日本語本文は、PowerShell 上に生の日本語文字列として直接書かず、Unicode escape・code point・UTF-8 byte 列などから Python 側で復元すること
- 置換対象は一意な ASCII プレースホルダ 1 箇所に限定し、前後の既存日本語本文をキーにした広い置換を避けること
- 追記後は、追加した見出しまたは節の末尾まで再読込し、重複・欠落・`?` 混入がないことを確認すること

## 日本語ファイル編集時の必須確認手順

日本語を含むファイルを編集する場合、編集前後に必ず以下を行うこと。

1. UTF-8 として再読込する
2. 編集対象の前後 20〜50 行を確認する
3. 編集した見出しの末尾まで確認する
4. `git diff --word-diff` または通常 diff で差分を確認する
5. `?`、文字化け、見出し重複、箇条書き重複がないか確認する
6. 必要なら `rg "�|\?" <file>` 等で異常文字や `?` を確認する

## 日本語編集で異常を検知した場合

編集後に文字化け、見出し重複、断片の欠落、想定外の大量差分を検知した場合は、そのまま編集を続けず停止すること。
停止後は、

- どのファイルで
- どの操作後に
- 何が起きたか
を簡潔に報告すること。

## 日本語文字列を含む置換の原則

日本語文字列をキーにした一括置換は、対象が一意に特定できることを確認してから行うこと。
同一文字列が複数存在する可能性がある場合は、

- 見出し
- 前後文脈
- 開始位置と終了位置
を確認し、狭い範囲に限定して編集すること。

## 変更後の必須確認

変更を加えたら、commit前に最低限以下を確認すること。

- api: PHPコード整形と軽いテストを通す
- web: lint と format check を通す
- docs/api/openapi.yaml を変更した場合は YAML と整合を確認する
- docs / OpenAPI / 実装 / test のいずれかを変えたら、関連差分の更新漏れがないか確認する
- commit やコミットメッセージ提案の前に、`.githooks/pre-commit` の実行内容を確認し、必要なら同等コマンドを先に実行する
- web に変更がある場合、`npm run lint` だけでなく `npx prettier --check .` も事前に実行する
- seed / category master / migration / test dataset に関わる変更では、テストコマンド案内時に `php artisan migrate` / `php artisan db:seed --class=TestDatasetSeeder` または `php artisan migrate:fresh --seed` の要否もあわせて案内する
- 整形コマンドで修正が入った場合は、その差分を再 stage してから commit する
- bash / WSL 側で `node` や `npx` の実行結果が不安定な場合は、Windows 側の `cmd.exe /c` で `npm run lint` と `npx prettier --check .` を確認する
- `prettier --write` を実行した後は、index と working tree がずれるため、対象ファイルを必ず `git add` し直してから commit する
- pre-commit で `Code style issues found` が出た場合は、まず hook が指摘したファイルに対して `prettier --check` を再現し、必要ならそのファイル群だけ `prettier --write` して再 stage する

変更範囲に関係ない大規模修正は行わないこと。

## 主要確認コマンド

- api: `php vendor/bin/pint --test`
- api: `php artisan test`
- web: `npm run lint`
- web: `npx prettier --check .`
- web (Windows 側確認): `cmd.exe /c "cd /d C:\DEV\wardrobe-app\web && npm run lint"`
- web (Windows 側確認): `cmd.exe /c "cd /d C:\DEV\wardrobe-app\web && npx prettier --check ."`

## Git hooks

- このリポジトリでは `core.hooksPath=.githooks` を使用する
- commit 前チェックは `.githooks/pre-commit` を前提とする
- `.githooks/pre-commit` は web 変更時に staged files ではなく `web` 全体へ `npx prettier --check .` を実行する前提で扱う
- hook の変更時は README または docs のセットアップ手順も必要に応じて更新する
- pre-commit は repo 全体の既存負債ではなく、原則として今回変更した範囲を中心に確認する
